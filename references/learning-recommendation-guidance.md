# Learning Recommendation Guidance

Learning recommendations should help the team grow together. They should not sound like remediation for poor performers.

## Recommended Formats

- short workshop
- refactoring kata
- guided code review
- before/after walkthrough
- paired remediation
- architecture decision review
- standard plus examples
- brown-bag discussion

## Match Format to Problem

| Pattern | Good learning format |
|---|---|
| Anemic domain model | Refactoring kata using a real but sanitized example |
| Controller/service responsibility confusion | Layered architecture walkthrough |
| Weak error semantics | Failure modeling workshop |
| Message consumer reliability gaps | Message-processing reliability design session |
| Reflection-heavy tests | Design-for-testability kata |
| Runtime interpreter replacing typed models | Typed mapper design comparison |
| Quality exclusions and suppressions | Quality-gate configuration review |

## Tone

Use language like:

- "The assessments suggest a team-wide opportunity to strengthen..."
- "A workshop would help create shared judgment around..."
- "This is a good candidate for a before/after refactoring example..."

Avoid language like:

- "The team does not understand..."
- "Developers failed to..."
- "This proves incompetence..."
